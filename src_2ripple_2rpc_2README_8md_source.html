12 # How to use RPC coroutines.
13 
14 ## Introduction.
15 
16 By default, an RPC handler runs as an uninterrupted task on the JobQueue. This
17 is fine for commands that are fast to compute but might not be acceptable for
18 tasks that require multiple parts or are large, like a full ledger.
19 
20 For this purpose, the rippled RPC handler allows *suspension with continuation*
21 - a request to suspend execution of the RPC response and to continue it after
22 some function or job has been executed. A default continuation is supplied
23 which simply reschedules the job on the JobQueue, or the programmer can supply
24 their own.
25 
26 ## The classes.
27 
28 Suspension with continuation uses four `std::function`s in the `ripple::RPC`
29 namespace:
30 
31  using Callback = std::function <void ()>;
32  using Continuation = std::function <void (Callback const&)>;
33  using Suspend = std::function <void (Continuation const&)>;
34  using Coroutine = std::function <void (Suspend const&)>;
35 
36 A `Callback` is a generic 0-argument function. A given `Callback` might or might
37 not block. Unless otherwise advised, do not hold locks or any resource that
38 would prevent any other task from making forward progress when you call a
39 `Callback`.
40 
41 A `Continuation` is a function that is given a `Callback` and promises to call
42 it later. A `Continuation` guarantees to call the `Callback` exactly once at
43 some point in the future, but it does not have to be immediately or even in the
44 current thread.
45 
46 A `Suspend` is a function belonging to a `Coroutine`. A `Suspend` runs a
47 `Continuation`, passing it a `Callback` that continues execution of the
48 `Coroutine`.
49 
50 And finally, a `Coroutine` is a `std::function` which is given a
51 `Suspend`. This is what the RPC handler gives to the coroutine manager,
52 expecting to get called back with a `Suspend` and to be able to start execution.
53 
54 ## The flow of control.
55 
56 Given these functions, the flow of RPC control when using coroutines is
57 straight-forward.
58 
59 1. The instance of `ServerHandler` receives an RPC request.
60 
61 2. It creates a `Coroutine` and gives it to the coroutine manager.
62 
63 3. The coroutine manager creates a `Coroutine`, starts it up, and then calls
64  the `Coroutine` with a `Suspend`.
65 
66 4. Now the RPC response starts to be calculated.
67 
68 5. When the RPC handler wants to suspend, it calls the `Suspend` function with
69  a `Continuation`.
70 
71 6. Coroutine execution is suspended.
72 
73 7. The `Continuation` is called with a `Callback` that the coroutine manager
74  creates.
75 
76 8. The `Continuation` may choose to execute immediately, defer execution on the
77  job queue, or wait for some resource to be free.
78 
79 9. When the `Continuation` is finished, it calls the `Callback` that the
80  coroutine manager gave it, perhaps a long time ago.
81 
82 10. This `Callback` continues execution on the suspended `Coroutine` from where
83  it left off.
