1 # Coding Standards
2 
3 Coding standards used here gradually evolve and propagate through
4 code reviews. Some aspects are enforced more strictly than others.
5 
6 ## Rules
7 
8 These rules only apply to our own code. We can't enforce any sort of
9 style on the external repositories and libraries we include. The best
10 guideline is to maintain the standards that are used in those libraries.
11 
12 * Tab inserts 4 spaces. No tab characters.
13 * Braces are indented in the [Allman style][1].
14 * Modern C++ principles. No naked ```new``` or ```delete```.
15 * Line lengths limited to 80 characters. Exceptions limited to data and tables.
16 
17 ## Guidelines
18 
19 If you want to do something contrary to these guidelines, understand
20 why you're doing it. Think, use common sense, and consider that this
21 your changes will probably need to be maintained long after you've
22 moved on to other projects.
23 
24 * Use white space and blank lines to guide the eye and keep your intent clear.
25 * Put private data members at the top of a class, and the 6 public special
26 members immediately after, in the following order:
27  * Destructor
28  * Default constructor
29  * Copy constructor
30  * Copy assignment
31  * Move constructor
32  * Move assignment
33 * Don't over-inline by defining large functions within the class
34 declaration, not even for template classes.
35 
36 ## Formatting
37 
38 The goal of source code formatting should always be to make things as easy to
39 read as possible. White space is used to guide the eye so that details are not
40 overlooked. Blank lines are used to separate code into "paragraphs."
41 
42 * Always place a space before and after all binary operators,
43  especially assignments (`operator=`).
44 * The `!` operator should be preceded by a space, but not followed by one.
45 * The `~` operator should be preceded by a space, but not followed by one.
46 * The `++` and `--` operators should have no spaces between the operator and
47  the operand.
48 * A space never appears before a comma, and always appears after a comma.
49 * Don't put spaces after a parenthesis. A typical member function call might
50  look like this: `foobar (1, 2, 3);`
51 * In general, leave a blank line before an `if` statement.
52 * In general, leave a blank line after a closing brace `}`.
53 * Do not place code on the same line as any opening or
54  closing brace.
55 * Do not write `if` statements all-on-one-line. The exception to this is when
56  you've got a sequence of similar `if` statements, and are aligning them all
57  vertically to highlight their similarities.
58 * In an `if-else` statement, if you surround one half of the statement with
59  braces, you also need to put braces around the other half, to match.
60 * When writing a pointer type, use this spacing: `SomeObject* myObject`.
61  Technically, a more correct spacing would be `SomeObject *myObject`, but
62  it makes more sense for the asterisk to be grouped with the type name,
63  since being a pointer is part of the type, not the variable name. The only
64  time that this can lead to any problems is when you're declaring multiple
65  pointers of the same type in the same statement - which leads on to the next
66  rule:
67 * When declaring multiple pointers, never do so in a single statement, e.g.
68  `SomeObject* p1, *p2;` - instead, always split them out onto separate lines
69  and write the type name again, to make it quite clear what's going on, and
70  avoid the danger of missing out any vital asterisks.
71 * The previous point also applies to references, so always put the `&` next to
72  the type rather than the variable, e.g. `void foo (Thing const& thing)`. And
73  don't put a space on both sides of the `*` or `&` - always put a space after
74  it, but never before it.
75 * The word `const` should be placed to the right of the thing that it modifies,
76  for consistency. For example `int const` refers to an int which is const.
77  `int const*` is a pointer to an int which is const. `int *const` is a const
78  pointer to an int.
79 * Always place a space in between the template angle brackets and the type
80  name. Template code is already hard enough to read!
