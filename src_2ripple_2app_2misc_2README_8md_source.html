1 # Fee Voting
2 
3 The Ripple payment protocol enforces a fee schedule expressed in units of the
4 native currency, XRP. Fees for transactions are paid directly from the account
5 owner. There are also reserve requirements for each item that occupies storage
6 in the ledger. The reserve fee schedule contains both a per-account reserve,
7 and a per-owned-item reserve. The items an account may own include active
8 offers, trust lines, and tickets.
9 
10 Validators may vote to increase fees if they feel that the network is charging
11 too little. They may also vote to decrease fees if the fees are too costly
12 relative to the value the network provides. One common case where a validator
13 may want to change fees is when the value of the native currency XRP fluctuates
14 relative to other currencies.
15 
16 The fee voting mechanism takes place every 256 ledgers ("voting ledgers"). In
17 a voting ledger, each validator takes a position on what they think the fees
18 should be. The consensus process converges on the majority position, and in
19 subsequent ledgers a new fee schedule is enacted.
20 
21 ## Consensus
22 
23 The Ripple consensus algorithm allows distributed participants to arrive at
24 the same answer for yes/no questions. The canonical case for consensus is
25 whether or not a particular transaction is included in the ledger. Fees
26 present a more difficult challenge, since the decision on the new fee is not
27 a yes or no question.
28 
29 To convert validators' positions on fees into a yes or no question that can
30 be converged in the consensus process, the following algorithm is used:
31 
32 - In the ledger before a voting ledger, validators send proposals which also
33  include the values they think the network should use for the new fee schedule.
34 
35 - In the voting ledger, validators examine the proposals from other validators
36  and choose a new fee schedule which moves the fees in a direction closer to
37  the validator's ideal fee schedule and is also likely to be accepted. A fee
38  amount is likely to be accepted if a majority of validators agree on the
39  number.
40 
41 - Each validator injects a "pseudo transaction" into their proposed ledger
42  which sets the fees to the chosen schedule.
43 
44 - The consensus process is applied to these fee-setting transactions as normal.
45  Each transaction is either included in the ledger or not. In most cases, one
46  fee setting transaction will make it in while the others are rejected. In
47  some rare cases more than one fee setting transaction will make it in. The
48  last one to be applied will take effect. This is harmless since a majority
49  of validators still agreed on it.
50 
51 - After the voting ledger has been validated, future pseudo transactions
52  before the next voting ledger are rejected as fee setting transactions may
53  only appear in voting ledgers.
54 
55 ## Configuration
56 
57 A validating instance of rippled uses information in the configuration file
58 to determine how it wants to vote on the fee schedule. It is the responsibility
59 of the administrator to set these values.

63 # Amendment
64 
65 An Amendment is a new or proposed change to a ledger rule. Ledger rules affect
66 transaction processing and consensus; peers must use the same set of rules for
67 consensus to succeed, otherwise different instances of rippled will get
68 different results. Amendments can be almost anything but they must be accepted
69 by a network majority through a consensus process before they are utilized. An
70 Amendment must receive at least an 80% approval rate from validating nodes for
71 a period of two weeks before being accepted. The following example outlines the
72 process of an Amendment from its conception to approval and usage.
73 
74 * A community member makes proposes to change transaction processing in some
75  way. The proposal is discussed amongst the community and receives its support
76  creating a community or human consensus.
77 
78 * Some members contribute their time and work to develop the Amendment.
79 
80 * A pull request is created and the new code is folded into a rippled build
81  and made available for use.
82 
83 * The consensus process begins with the validating nodes.
84 
85 * If the Amendment holds an 80% majority for a two week period, nodes will begin
86  including the transaction to enable it in their initial sets.
87 
88 Nodes may veto Amendments they consider undesirable by never announcing their
89 support for those Amendments. Just a few nodes vetoing an Amendment will normally
90 keep it from being accepted. Nodes could also vote yes on an Amendments even
91 before it obtains a super-majority. This might make sense for a critical bug fix.
92 
93 Validators that support an amendment that is not yet enabled announce their
94 support in their validations. If 80% support is achieved, they will introduce
95 a pseudo-transaction to track the amendment's majority status in the ledger.
96 
97 If an amendment whose majority status is reported in a ledger loses that
98 majority status, validators will introduce pseudo-transactions to remove
99 the majority status from the ledger.
100 
101 If an amendment holds majority status for two weeks, validators will
102 introduce a pseudo-transaction to enable the amendment.
103 
104 All amednements are assumed to be critical and irreversible. Thus there
105 is no mechanism to disable or revoke an amendment, nor is there a way
106 for a server to operate while an amendment it does not understand is
107 enabled.

111 # SHAMapStore: Online Delete
112 
113 Optional online deletion happens through the SHAMapStore. Records are deleted
114 from disk based on ledger sequence number. These records reside in the
115 key-value database as well as in the SQLite ledger and transaction databases.
116 Without online deletion storage usage grows without bounds. It can only
117 be pruned by stopping, manually deleting data, and restarting the server.
118 Online deletion requires less operator intervention to manage the server.
119 
120 The main mechanism to delete data from the key-value database is to keep two
121 databases open at all times. One database has all writes directed to it. The
122 other database has recent archival data from just prior to that from the current
123 writable database.
124 Upon rotation, the archival database is deleted. The writable database becomes
125 archival, and a brand new database becomes writable. To ensure that no
126 necessary data for transaction processing is lost, a variety of steps occur,
127 including copying the contents of an entire ledger's account state map,
128 clearing caches, and copying the contents of (freshening) other caches.
129 
130 Deleting from SQLite involves more straight-forward SQL DELETE queries from
131 the respective tables, with a rudimentary back-off algorithm to do portions
132 of the deletions at a time. This back-off is in place so that the database
133 lock is not held excessively. The SQLite database is not configured to
134 delete on-disk storage, so it will grow over time. However, with online delete
135 enabled, it grows at a very small rate compared with the key-value store.
136 
137 The online delete routine aborts its current activities if it fails periodic
138 server health checks. This minimizes impact of I/O and locking of critical
139 objects. If interrupted, the routine will start again at the next validated
140 ledger close. Likewise, the routine will continue in a similar fashion if the
141 server restarts.
142 
143 Configuration:
144 
145 * In the [node_db] configuration section, an optional online_delete parameter is
146 set. If not set or if set to 0, online delete is disabled. Otherwise, the
147 setting defines number of ledgers between deletion cycles.
148 * Another optional parameter in [node_db] is that for advisory_delete. It is
149 disabled by default. If set to non-zero, requires an RPC call to activate the
150 deletion routine.
151 * online_delete must not be greater than the [ledger_history] parameter.
152 * [fetch_depth] will be silently set to equal the online_delete setting if
153 online_delete is greater than fetch_depth.
154 * In the [node_db] section, there is a performance tuning option, delete_batch,
155 which sets the maximum size in ledgers for each SQL DELETE query.
