3 ## Introduction
4 
5 The _XRP Ledger network_ consists of a collection of _peers_ running
6 **`rippled`** or other compatible software. Each peer maintains multiple
7 outgoing connections and optional incoming connections to other peers.
8 These connections are made over both the public Internet and private local
9 area networks. This network defines a connected directed graph of nodes
10 where vertices are instances of `rippled` and edges are persistent TCP/IP
11 connections. Peers send and receive messages to other connected peers. This
12 peer to peer network, layered on top of the public and private Internet,
13 forms an [_overlay network_][overlay_network]. The contents of the messages
14 and the behavior of peers in response to the messages, plus the information
15 exchanged during the handshaking phase of connection establishment, defines
16 the _XRP Ledger peer protocol_ (or _protocol_ in this context).
17 
18 ## Overview
19 
20 Each connection is represented by a _Peer_ object. The Overlay manager
21 establishes, receives, and maintains connections to peers. Protocol
22 messages are exchanged between peers and serialized using
23 [_Google Protocol Buffers_][protocol_buffers].
24 
25 ### Structure
26 
27 Each connection between peers is identified by its connection type, which
28 affects the behavior of message routing. At present, only a single connection
29 type is supported: **Peer**.
30 
31 ## Handshake
32 
33 To establish a protocol connection, a peer makes an outgoing TLS encrypted
34 connection to a remote peer, then sends an HTTP request with no message body.
35 
36 ### HTTP
37 
38 The HTTP [request](https://www.w3.org/Protocols/rfc2616/rfc2616-sec5.html) must:
39 
40 - Use HTTP version 1.1.
41 - Specify a request URI consisting of a single forward slash character ("/")
42 indicating the server root. Requests using different URIs are reserved for
43 future protocol implementations.
44 - Use the [_HTTP/1.1 Upgrade_][upgrade_header] mechanism with additional custom
45 fields to communicate protocol specific information related to the upgrade.
46 
47 HTTP requests which do not conform to this requirements must generate an
48 appropriate HTTP error and result in the connection being closed.
49 
50 Upon receipt of a well-formed HTTP upgrade request, and validation of the
51 protocol specific parameters, a peer will either send back a HTTP 101 response
52 and switch to the requested protocol, or a message indicating that the request
53 failed (e.g. by sending HTTP 400 "Bad Request" or HTTP 503 "Service Unavailable").
54

303 ### Session Signature
304 
305 Even for SSL/TLS encrypted connections, it is possible for an attacker to mount
306 relatively inexpensive MITM attacks that can be extremely hard to detect and
307 may afford the attacker the ability to intelligently tamper with messages
308 exchanged between the two endpoints.
309 
310 This risk can be mitigated if at least one side has a certificate from a certificate
311 authority trusted by the other endpoint, but having a certificate is not always
312 possible (or even desirable) in a decentralized and permissionless network.
313 
314 Ultimately, the goal is to ensure that two endpoints A and B know that they are
315 talking directly to each other over a single end-to-end SSL/TLS session instead
316 of two separate SSL/TLS sessions, with an attacker acting as a proxy.
317 
318 The XRP Ledger protocol prevents this attack by leveraging the fact that the two
319 servers each have a node identity, in the form of **`secp256k1`** keypairs, and
320 use that to strongly bind the SSL/TLS session to the node identities of each of
321 the two servers at the end of the SSL/TLS session.
322 
323 To do this we "reach into" the SSL/TLS session, and extract the **`finished`**
324 messages for the local and remote endpoints, and combine them to generate a unique
325 "fingerprint". By design, this fingerprint should be the same for both SSL/TLS
326 endpoints.
327 
328 That fingerprint, which is never shared over the wire (since each endpoint will
329 calculate it independently), is then signed by each server using its public
330 **`secp256k1`** node identity and the signature is transferred over the SSL/TLS
331 encrypted link during the protocol handshake phase.
332 
333 Each side of the link will verify that the provided signature is from the claimed
334 public key against the session's unique fingerprint. If this signature check fails
335 then the link **MUST** be dropped.
336 
337 If an attacker, Eve, establishes two separate SSL sessions with Alice and Bob, the
338 fingerprints of the two sessions will be different, and Eve will not be able to
339 sign the fingerprint of her session with Bob with Alice's private key, or the
340 fingerprint of her session with Alice with Bob's private key, and so both A and
341 B will know that an active MITM attack is in progress and will close their
342 connections.
343 
344 If Eve simply proxies the raw bytes, she will be unable to decrypt the data being
345 transferred between A and B and will not be able to intelligently tamper with the
346 message stream between Alice and Bob, although she may be still be able to inject
347 delays or terminate the link.

350 # Ripple Clustering #
351 
352 A cluster consists of more than one Ripple server under common
353 administration that share load information, distribute cryptography
354 operations, and provide greater response consistency.
355 
356 Cluster nodes are identified by their public node keys. Cluster nodes
357 exchange information about endpoints that are imposing load upon them.
358 Cluster nodes share information about their internal load status. Cluster
359 nodes do not have to verify the cryptographic signatures on messages
360 received from other cluster nodes.
361 
362 ## Configuration ##
363 
364 A server's public key can be determined from the output of the `server_info`
365 command. The key is in the `pubkey_node` value, and is a text string
366 beginning with the letter `n`. The key is maintained across runs in a
367 database.
368 
369 Cluster members are configured in the `rippled.cfg` file under
370 `[cluster_nodes]`. Each member should be configured on a line beginning
371 with the node public key, followed optionally by a space and a friendly
372 name.
373 
374 Because cluster members can introduce other cluster members, it is not
375 necessary to configure every cluster member on every other cluster member.
376 If a hub and spoke system is used, it is sufficient to configure every
377 cluster member on the hub(s) and only configure the hubs on the spokes.
378 That is, each spoke does not need to be configured on every other spoke.
379 
380 New spokes can be added as follows:
381 
382 - In the new spoke's `[cluster_nodes]`, include each hub's public node key
383 - Start the spoke server and determine its public node key
384 - Configure each hub with the new spoke's public key
385 - Restart each hub, one by one
386 - Restart the spoke
387 
388 ## Transaction Behavior ##
389 
390 When a transaction is received from a cluster member, several normal checks
391 are bypassed:
392 
393 Signature checking is bypassed because we trust that a cluster member would
394 not relay a transaction with an incorrect signature. Validators may wish to
395 disable this feature, preferring the additional load to get the additional
396 security of having validators check each transaction.
397 
398 Local checks for transaction checking are also bypassed. For example, a
399 server will not reject a transaction from a cluster peer because the fee
400 does not meet its current relay fee. It is preferable to keep the cluster
401 in agreement and permit confirmation from one cluster member to more
402 reliably indicate the transaction's acceptance by the cluster.
403 
404 ## Server Load Information ##
405 
406 Cluster members exchange information on their server's load level. The load
407 level is essentially the amount by which the normal fee levels are multiplied
408 to get the server's fee for relaying transactions.
409 
410 A server's effective load level, and the one it uses to determine its relay
411 fee, is the highest of its local load level, the network load level, and the
412 cluster load level. The cluster load level is the median load level reported
413 by a cluster member.
414 
415 ## Gossip ##
416 
417 Gossip is the mechanism by which cluster members share information about
418 endpoints (typically IPv4 addresses) that are imposing unusually high load
419 on them. The endpoint load manager takes into account gossip to reduce the
420 amount of load the endpoint is permitted to impose on the local server
421 before it is warned, disconnected, or banned.
422 
423 Suppose, for example, that an attacker controls a large number of IP
424 addresses, and with these, he can send sufficient requests to overload a
425 server. Without gossip, he could use these same addresses to overload all
426 the servers in a cluster. With gossip, if he chooses to use the same IP
427 address to impose load on more than one server, he will find that the amount
428 of load he can impose before getting disconnected is much lower.
429 
430 ## Monitoring ##
431 
432 The `peers` command will report on the status of the cluster. The `cluster`
433 object will contain one entry for each member of the cluster (either configured
434 or introduced by another cluster member). The `age` field is the number of
435 seconds since the server was last heard from. If the server is reporting an
436 elevated cluster fee, that will be reported as well.
437 
438 In the `peers` object, cluster members will contain a `cluster` field set to `true`.
