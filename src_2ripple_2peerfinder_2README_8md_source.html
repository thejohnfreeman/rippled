2 # PeerFinder
3 
4 ## Introduction
5 
6 The _Ripple payment network_ consists of a collection of _peers_ running the
7 **rippled software**. Each peer maintains multiple outgoing connections and
8 optional incoming connections to other peers. These connections are made over
9 both the public Internet and private local area networks. This network defines
10 a fully connected directed graph of nodes. Peers send and receive messages to
11 other connected peers. This peer to peer network, layered on top of the public
12 and private Internet, forms an [_overlay network_][overlay_network].
13 
14 ## Bootstrapping
15 
16 When a peer comes online it needs a set of IP addresses to connect to in order to
17 gain initial entry into the overlay in a process called _bootstrapping_. Once they
18 have established an initial set of these outbound peer connections, they need to
19 gain additional addresses to establish more outbound peer connections until the
20 desired limit is reached. Furthermore, they need a mechanism to advertise their
21 IP address to new or existing peers in the overlay so they may receive inbound
22 connections up to some desired limit. And finally, they need a mechanism to provide
23 inbound connection requests with an alternate set of IP addresses to try when they
24 have already reached their desired maximum number of inbound connections.
25 
26 PeerFinder is a self contained module that provides these services, along with some
27 additional overlay network management services such as _fixed slots_ and _cluster
28 slots_.
29 
30 ## Features
31 
32 PeerFinder has these responsibilities
33 
34 * Maintain a persistent set of endpoint addresses suitable for bootstrapping
35  into the peer to peer overlay, ranked by relative locally observed utility.
36 
37 * Send and receive protocol messages for discovery of endpoint addresses.
38 
39 * Provide endpoint addresses to new peers that need them.
40 
41 * Maintain connections to a configured set of fixed peers.
42 
43 * Impose limits on the various slots consumed by peer connections.
44 
45 * Initiate outgoing connection attempts to endpoint addresses to maintain the
46  overlay connectivity and fixed peer policies.
47 
48 * Verify the connectivity of neighbors who advertise inbound connection slots.
49 
50 * Prevent duplicate connections and connections to self.

54 # Concepts
55 
56 ## Manager
57 
58 The `Manager` is an application singleton which provides the primary interface
59 to interaction with the PeerFinder.
60 
61 ### Autoconnect
62 
63 The Autoconnect feature of PeerFinder automatically establishes outgoing
64 connections using addresses learned from various sources including the
65 configuration file, the result of domain name lookups, and messages received
66 from the overlay itself.
67 
68 ### Callback
69 
70 PeerFinder is an isolated code module with few external dependencies. To perform
71 socket specific activities such as establishing outgoing connections or sending
72 messages to connected peers, the Manager is constructed with an abstract
73 interface called the `Callback`. An instance of this interface performs the
74 actual required operations, making PeerFinder independent of the calling code.
75 
76 ### Config
77 
78 The `Config` structure defines the operational parameters of the PeerFinder.
79 Some values come from the configuration file while others are calculated via
80 tuned heuristics. The fields are as follows:
81 
82 * `autoConnect`
83 
84  A flag indicating whether or not the Autoconnect feature is enabled.
85 
86 * `wantIncoming`
87 
88  A flag indicating whether or not the peer desires inbound connections. When
89  this flag is turned off, a peer will not advertise itself in Endpoint
90  messages.
91 
92 * `listeningPort`
93 
94  The port number to use when creating the listening socket for peer
95  connections.
96 
97 * `maxPeers`
98 
99  The largest number of active peer connections to allow. This includes inbound
100  and outbound connections, but excludes fixed and cluster peers. There is an
101  implementation defined floor on this value.
102 
103 * `outPeers`
104 
105  The number of automatic outbound connections that PeerFinder will maintain
106  when the Autoconnect feature is enabled. The value is computed with fractional
107  precision as an implementation defined percentage of `maxPeers` subject to
108  an implementation defined floor. An instance of the PeerFinder rounds the
109  fractional part up or down using a uniform random number generated at
110  program startup. This allows the outdegree of the overlay network to be
111  controlled with fractional precision, ensuring that all inbound network
112  connection slots are not consumed (which would make it difficult for new
113  participants to enter the network).

120 ### Livecache
121 
122 The Livecache holds relayed IP addresses that have been received recently in
123 the form of Endpoint messages via the peer to peer overlay. A peer periodically
124 broadcasts the Endpoint message to its neighbors when it has open inbound
125 connection slots. Peers store these messages in the Livecache and periodically
126 forward their neighbors a handful of random entries from their Livecache, with
127 an incremented hop count for each forwarded entry.
128 
129 The algorithm for sending a neighbor a set of Endpoint messages chooses evenly
130 from all available hop counts on each send. This ensures that each peer
131 will see some entries with the farthest hops at each iteration. The result is
132 to expand a peer's horizon with respect to which overlay endpoints are visible.
133 This is designed to force the overlay to become highly connected and reduce
134 the network diameter with each connection establishment.
135 
136 When a peer receives an Endpoint message that originates from a neighbor
137 (identified by a hop count of zero) for the first time, it performs an incoming
138 connection test on that neighbor by initiating an outgoing connection to the
139 remote IP address as seen on the connection combined with the port advertised
140 in the Endpoint message. If the test fails, then the peer considers its neighbor
141 firewalled (intentionally or due to misconfiguration) and no longer forwards
142 Endpoint messages for that peer. This prevents poor quality unconnectible
143 addresses from landing in the caches. If the incoming connection test passes,
144 then the peer fills in the Endpoint message with the remote address as seen on
145 the connection before storing it in its cache and forwarding it to other peers.
146 This relieves the neighbor from the responsibility of knowing its own IP address
147 before it can start receiving incoming connections.
148 
149 Livecache entries expire quickly. Since a peer stops advertising itself when
150 it no longer has available inbound slots, its address will shortly after stop
151 being handed out by other peers. Livecache entries are very likely to result
152 in both a successful connection establishment and the acquisition of an active
153 outbound slot. Compare this with Bootcache addresses, which are very likely to
154 be connectible but unlikely to have an open slot.
155 
156 Because entries in the Livecache are ephemeral, they are not persisted across
157 launches in the database. The Livecache is continually updated and expired as
158 Endpoint messages are received from the overlay over time.
159 
160 ### Bootcache
161 
162 The `Bootcache` stores IP addresses useful for gaining initial connections.
163 Each address is associated with the following metadata:
164 
165 * **Valence**
166 
167  A signed integer which represents the number of successful
168  consecutive connection attempts when positive, and the number of
169  failed consecutive connection attempts when negative. If an outgoing
170  connection attempt to the corresponding IP address fails to complete the
171  handshake the valence is reset to negative one. This harsh penalty is
172  intended to prevent popular servers from forever remaining top ranked in
173  all peer databases.
174 
175 When choosing addresses from the boot cache for the purpose of
176 establishing outgoing connections, addresses are ranked in decreasing order of
177 valence. The Bootcache is persistent. Entries are periodically inserted and
178 updated in the corresponding SQLite database during program operation. When
179 **rippled** is launched, the existing Bootcache database data is accessed and
180 loaded to accelerate the bootstrap process.
181 
182 Desirable entries in the Bootcache are addresses for servers which are known to
183 have high uptimes, and for which connection attempts usually succeed. However,
184 these servers do not necessarily have available inbound connection slots.
185 However, it is assured that these servers will have a well populated Livecache
186 since they will have moved towards the core of the overlay over their high
187 uptime. When a connected server is full it will return a handful of new
188 addresses from its Livecache and gracefully close the connection. Addresses
189 from the Livecache are highly likely to have inbound connection slots and be
190 connectible.
191 
192 For security, all information that contributes to the ranking of Bootcache
193 entries is observed locally. PeerFinder never trusts external sources of information.
194 
195 ### Slot
196 
197 Each TCP/IP socket that can participate in the peer to peer overlay occupies
198 a slot. Slots have properties and state associated with them:
199 
200 #### State (Slot)
201 
202 The slot state represents the current stage of the connection as it passes
203 through the business logic for establishing peer connections.
204 
205 * `accept`
206 
207  The accept state is an initial state resulting from accepting an incoming
208  connection request on a listening socket. The remote IP address and port
209  are known, and a handshake is expected next.
210 
211 * `connect`
212 
213  The connect state is an initial state used when actively establishing outbound
214  connection attempts. The desired remote IP address and port are known.
215 
216 * `connected`
217 
218  When an outbound connection attempt succeeds, it moves to the connected state.
219  The handshake is initiated but not completed.
220 
221 * `active`
222 
223  The state becomes Active when a connection in either the Accepted or Connected
224  state completes the handshake process, and a slot is available based on the
225  properties. If no slot is available when the handshake completes, the socket
226  is gracefully closed.
227 
228 * `closing`
229 
230  The Closing state represents a connected socket in the process of being
231  gracefully closed.
232 
233 #### Properties (Slot)
234 
235 Slot properties may be combined and are not mutually exclusive.
236 
237 * **Inbound**
238 
239  An inbound slot is the condition of a socket which has accepted an incoming
240  connection request. A connection which is not inbound is by definition
241  outbound.
242 
243 * **Fixed**
244 
245  A fixed slot is a desired connection to a known peer identified by IP address,
246  usually entered manually in the configuration file. For the purpose of
247  establishing outbound connections, the peer also has an associated port number
248  although only the IP address is checked to determine if the fixed peer is
249  already connected. Fixed slots do not count towards connection limits.
250 
251 * **Cluster**
252 
253  A cluster slot is a connection which has completed the handshake stage, whose
254  public key matches a known public key usually entered manually in the
255  configuration file or learned through overlay messages from other trusted
256  peers. Cluster slots do not count towards connection limits.
257 
258 * **Superpeer** (forthcoming)
259 
260  A superpeer slot is a connection to a peer which can accept incoming
261  connections, meets certain resource availaibility requirements (such as
262  bandwidth, CPU, and storage capacity), and operates full duplex in the
263  overlay. Connections which are not superpeers are by definition leaves. A
264  leaf slot is a connection to a peer which does not route overlay messages to
265  other peers, and operates in a partial half duplex fashion in the overlay.
266 
267 #### Fixed Slots
268 
269 Fixed slots are identified by IP address and set up during the initialization
270 of the Manager, usually from the configuration file. The Logic will always make
271 outgoing connection attempts to each fixed slot which is not currently
272 connected. If we receive an inbound connection from an endpoint whose address
273 portion (without port) matches a fixed slot address, we consider the fixed
274 slot to be connected.
275 
276 #### Cluster Slots
277 
278 Cluster slots are identified by the public key and set up during the
279 initialization of the manager or discovered upon receipt of messages in the
280 overlay from trusted connections.

284 # Algorithms
285 
286 ## Connection Strategy
287 
288 The _Connection Strategy_ applies the configuration settings to establish
289 desired outbound connections. It runs periodically and progresses through a
290 series of stages, remaining in each stage until a condition is met
291 
292 ### Stage 1: Fixed Slots
293 
294 This stage is invoked when the number of active fixed connections is below the
295 number of fixed connections specified in the configuration, and one of the
296 following is true:
297 
298 * There are eligible fixed addresses to try
299 * Any outbound connection attempts are in progress
300 
301 Each fixed address is associated with a retry timer. On a fixed connection
302 failure, the timer is reset so that the address is not tried for some amount
303 of time, which increases according to a scheduled sequence up to some maximum
304 which is currently set to approximately one hour between retries. A fixed
305 address is considered eligible if we are not currently connected or attempting
306 the address, and its retry timer has expired.
307 
308 The PeerFinder makes its best effort to become fully connected to the fixed
309 addresses specified in the configuration file before moving on to establish
310 outgoing connections to foreign peers. This security feature helps rippled
311 establish itself with a trusted set of peers first before accepting untrusted
312 data from the network.
313 
314 ### Stage 2: Livecache
315 
316 The Livecache is invoked when Stage 1 is not active, autoconnect is enabled,
317 and the number of active outbound connections is below the number desired. The
318 stage remains active while:
319 
320 * The Livecache has addresses to try
321 * Any outbound connection attempts are in progress
322 
323 PeerFinder makes its best effort to exhaust addresses in the Livecache before
324 moving on to the Bootcache, because Livecache addresses are highly likely
325 to be connectible (since they are known to have been online within the last
326 minute), and highly likely to have an open slot for an incoming connection
327 (because peers only advertise themselves in the Livecache when they have
328 open slots).
329 
330 ### Stage 3: Bootcache
331 
332 The Bootcache is invoked when Stage 1 and Stage 2 are not active, autoconnect
333 is enabled, and the number of active outbound connections is below the number
334 desired. The stage remains active while:
335 
336 * There are addresses in the cache that have not been tried recently.
337 
338 Entries in the Bootcache are ranked, with highly connectible addresses preferred
339 over others. Connection attempts to Bootcache addresses are very likely to
340 succeed but unlikely to produce an active connection since the peers likely do
341 not have open slots. Before the remote peer closes the connection it will send
342 a handful of addresses from its Livecache to help the new peer coming online
343 obtain connections.
