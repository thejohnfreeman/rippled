3 ## Introduction
4 
5 A `NodeObject` is a simple object that the Ledger uses to store entries. It is
6 comprised of a type, a hash and a blob. It can be uniquely
7 identified by the hash, which is a 256 bit hash of the blob. The blob is a
8 variable length block of serialized data. The type identifies what the blob
9 contains. The fields are as follows:
10 
11 * `mType`
12 
13  An enumeration that determines what the blob holds. There are four
14  different types of objects stored.
15 
16  * **ledger**
17 
18  A ledger header.
19 
20  * **transaction**
21 
22  A signed transaction.
23 
24  * **account node**
25 
26  A node in a ledger's account state tree.
27 
28  * **transaction node**
29 
30  A node in a ledger's transaction tree.
31 
32 * `mHash`
33 
34  A 256-bit hash of the blob.
35 
36 * `mData`
37 
38  A blob containing the payload. Stored in the following format.
39 
40 |Byte | | |
41 |:------|:--------------------|:-------------------------|
42 |0...7 |unused | |
43 |8 |type |NodeObjectType enumeration|
44 |9...end|data |body of the object data |
45 ---
46 The `NodeStore` provides an interface that stores, in a persistent database, a
47 collection of NodeObjects that rippled uses as its primary representation of
48 ledger entries. All ledger entries are stored as NodeObjects and as such, need
49 to be persisted between launches. If a NodeObject is accessed and is not in
50 memory, it will be retrieved from the database.
51 
52 ## Backend
53 
54 The `NodeStore` implementation provides the `Backend` abstract interface,
55 which lets different key/value databases to be chosen at run-time. This allows
56 experimentation with different engines. Improvements in the performance of the
57 NodeStore are a constant area of research. The database can be specified in
58 the configuration file [node_db] section as follows.
59 
60 One or more lines of key / value pairs
61 
62 Example:
63 ```
64 type=RocksDB
65 path=rocksdb
66 compression=1
67 ```
68 Choices for 'type' (not case-sensitive)
69 
70 * **HyperLevelDB**
71 
72  An improved version of LevelDB (preferred).
73 
74 * **LevelDB**
75 
76  Google's LevelDB database (deprecated).
77 
78 * **none**
79 
80  Use no backend.
81 
82 * **RocksDB**
83 
84  Facebook's RocksDB database, builds on LevelDB.
85 
86 * **SQLite**
87 
88  Use SQLite.
89 
90 'path' speficies where the backend will store its data files.
91 
92 Choices for 'compression'
93 
94 * **0** off
95 
96 * **1** on (default)
