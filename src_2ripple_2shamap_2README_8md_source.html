5 The SHAMap is a Merkle tree (http://en.wikipedia.org/wiki/Merkle_tree).
6 The SHAMap is also a radix tree of radix 16
7 (http://en.wikipedia.org/wiki/Radix_tree).
8 
9 *We need some kind of sensible summary of the SHAMap here.*
10 
11 A given SHAMap always stores only one of three kinds of data:
12 
13  * Transactions with metadata
14  * Transactions without metadata, or
15  * Account states.
16 
17 So all of the leaf nodes of a particular SHAMap will always have a uniform
18 type. The inner nodes carry no data other than the hash of the nodes
19 beneath them.

22 ## SHAMap Types ##
23 
24 There are two different ways of building and using a SHAMap:
25 
26  1. A mutable SHAMap and
27  2. An immutable SHAMap
28 
29 The distinction here is not of the classic C++ immutable-means-unchanging
30 sense. An immutable SHAMap contains *nodes* that are immutable. Also,
31 once a node has been located in an immutable SHAMap, that node is
32 guaranteed to persist in that SHAMap for the lifetime of the SHAMap.
33 
34 So, somewhat counter-intuitively, an immutable SHAMap may grow as new nodes
35 are introduced. But an immutable SHAMap will never get smaller (until it
36 entirely evaporates when it is destroyed). Nodes, once introduced to the
37 immutable SHAMap, also never change their location in memory. So nodes in
38 an immutable SHAMap can be handled using raw pointers (if you're careful).
39 
40 One consequence of this design is that an immutable SHAMap can never be
41 "trimmed". There is no way to identify unnecessary nodes in an immutable
42 SHAMap that could be removed. Once a node has been brought into the
43 in-memory SHAMap, that node stays in memory for the life of the SHAMap.
44 
45 Most SHAMaps are immutable, in the sense that they don't modify or remove
46 their contained nodes.
47 
48 An example where a mutable SHAMap is required is when we want to apply
49 transactions to the last closed ledger. To do so we'd make a mutable
50 snapshot of the state tree and then start applying transactions to it.
51 Because the snapshot is mutable, changes to nodes in the snapshot will not
52 affect nodes in other SHAMAps.
53 
54 An example using a immutable ledger would be when there's an open ledger
55 and some piece of code wishes to query the state of the ledger. In this
56 case we don't wish to change the state of the SHAMap, so we'd use an
57 immutable snapshot.

60 ## SHAMap Creation ##
61 
62 A SHAMap is usually not created from vacuum. Once an initial SHAMap is
63 constructed, later SHAMaps are usually created by calling
64 snapShot(bool isMutable) on the original SHAMap(). The returned SHAMap
65 has the expected characteristics (mutable or immutable) based on the passed
66 in flag.
67 
68 It is cheaper to make an immutable snapshot of a SHAMap than to make a mutable
69 snapshot. If the SHAMap snapshot is mutable then any of the nodes that might
70 be modified must be copied before they are placed in the mutable map.

73 ## SHAMap Thread Safety ##
74 
75 *This description is obsolete and needs to be rewritten.*
76 
77 SHAMaps can be thread safe, depending on how they are used. The SHAMap
78 uses a SyncUnorderedMap for its storage. The SyncUnorderedMap has three
79 thread-safe methods:
80 
81  * size(),
82  * canonicalize(), and
83  * retrieve()
84 
85 As long as the SHAMap uses only those three interfaces to its storage
86 (the mTNByID variable [which stands for Tree Node by ID]) the SHAMap is
87 thread safe.

96 ## Late-arriving Nodes ##
97 
98 As we noted earlier, SHAMaps (even immutable ones) may grow. If a SHAMap
99 is searching for a node and runs into an empty spot in the tree, then the
100 SHAMap looks to see if the node exists but has not yet been made part of
101 the map. This operation is performed in the `SHAMap::fetchNodeExternalNT()`
102 method. The *NT* is this case stands for 'No Throw'.
103 
104 The `fetchNodeExternalNT()` method goes through three phases:
105 
106  1. By calling `getCache()` we attempt to locate the missing node in the
107  TreeNodeCache. The TreeNodeCache is a cache of immutable
108  SHAMapTreeNodes that are shared across all SHAMaps.
109 
110  Any SHAMapTreeNode that is immutable has a sequence number of zero.
111  When a mutable SHAMap is created then its SHAMapTreeNodes are given
112  non-zero sequence numbers. So the `assert (ret->getSeq() == 0)`
113  simply confirms that the TreeNodeCache indeed gave us an immutable node.
114 
115  2. If the node is not in the TreeNodeCache, we attempt to locate the node
116  in the historic data stored by the data base. The call to
117  to `fetch(hash)` does that work for us.
118 
119  3. Finally, if ledgerSeq_ is non-zero and we did't locate the node in the
120  historic data, then we call a MissingNodeHandler.
121 
122  The non-zero ledgerSeq_ indicates that the SHAMap is a complete map that
123  belongs to a historic ledger with the given (non-zero) sequence number.
124  So, if all expected data is always present, the MissingNodeHandler should
125  never be executed.
126 
127  And, since we now know that this SHAMap does not fully represent
128  the data from that ledger, we set the SHAMap's sequence number to zero.
129 
130 If phase 1 returned a node, then we already know that the node is immutable.
131 However, if either phase 2 executes successfully, then we need to turn the
132 returned node into an immutable node. That's handled by the call to
133 `make_shared<SHAMapTreeNode>` inside the try block. That code is inside
134 a try block because the `fetchNodeExternalNT` method promises not to throw.
135 In case the constructor called by `make_shared` throws we don't want to
136 break our promise.

139 ## Canonicalize ##
140 
141 The calls to `canonicalize()` make sure that if the resulting node is already
142 in the SHAMap, then we return the node that's already present -- we never
143 replace a pre-existing node. By using `canonicalize()` we manage a thread
144 race condition where two different threads might both recognize the lack of a
145 SHAMapTreeNode at the same time. If they both attempt to insert the node
146 then `canonicalize` makes sure that the first node in wins and the slower
147 thread receives back a pointer to the node inserted by the faster thread.
148 
149 There's a problem with the current SHAMap design that `canonicalize()`
150 accommodates. Two different trees can have the exact same node (the same
151 hash value) with two different IDs. If the TreeNodeCache returns a node
152 with the same hash but a different ID, then we assume that the ID of the
153 passed-in node is 'better' than the older ID in the TreeNodeCache. So we
154 construct a new SHAMapTreeNode by copying the one we found in the
155 TreeNodeCache, but we give the new node the new ID. Then we replace the
156 SHAMapTreeNode in the TreeNodeCache with this newly constructed node.
157 
158 The TreeNodeCache is not subject to the rule that any node must be
159 resident forever. So it's okay to replace the old node with the new node.
160 
161 The `SHAMap::getCache()` method exhibits the same behavior.

164 ## SHAMap Improvements ##
165 
166 Here's a simple one: the SHAMapTreeNode::mAccessSeq member is currently not
167 used and could be removed.
168 
169 Here's a more important change. The tree structure is currently embedded
170 in the SHAMapTreeNodes themselves. It doesn't have to be that way, and
171 that should be fixed.
172 
173 When we navigate the tree (say, like `SHAMap::walkTo()`) we currently
174 ask each node for information that we could determine locally. We know
175 the depth because we know how many nodes we have traversed. We know the
176 ID that we need because that's how we're steering. So we don't need to
177 store the ID in the node. The next refactor should remove all calls to
178 `SHAMapTreeNode::GetID()`.
179 
180 Then we can remove the NodeID member from SHAMapTreeNode.
181 
182 Then we can change the SHAMap::mTNBtID member to be mTNByHash.
183 
184 An additional possible refactor would be to have a base type, SHAMapTreeNode,
185 and derive from that InnerNode and LeafNode types. That would remove
186 some storage (the array of 16 hashes) from the LeafNodes. That refactor
187 would also have the effect of simplifying methods like `isLeaf()` and
188 `hasItem()`.
