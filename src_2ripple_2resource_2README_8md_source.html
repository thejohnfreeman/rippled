1 # Resource::Manager #
2 
3 The ResourceManager module has these responsibilities:
4 
5 - Uniquely identify endpoints which impose load.
6 - Track the load used across endpoints.
7 - Provide an interface to share load information in a cluster.
8 - Warn and/or disconnect endpoints for imposing load.
9 
10 ## Description ##
11 
12 To prevent monopolization of server resources or attacks on servers,
13 resource consumption is monitored at each endpoint. When consumption
14 exceeds certain thresholds, costs are imposed. Costs could include charging
15 additional XRP for transactions, requiring a proof of work to be
16 performed, or simply disconnecting the endpoint.
17 
18 Currently, consumption endpoints include websocket connections used to
19 service clients, and peer connections used to create the peer to peer
20 overlay network implementing the Ripple protocol.
21 
22 The current "balance" of a Consumer represents resource consumption
23 debt or credit. Debt is accrued when bad loads are imposed. Credit is
24 granted when good loads are imposed. When the balance crosses heuristic
25 thresholds, costs are increased on the endpoint. The balance is
26 represented as a unitless relative quantity. This balance is currently
27 held by the Entry struct in the impl/Entry.h file.
28 
29 Costs associated with specific transactions are defined in the
30 impl/Fees files.
31 
32 Although RPC connections consume resources, they are transient and
33 cannot be rate limited. It is advised not to expose RPC interfaces
34 to the general public.
35 
36 ## Consumer Types ##
37 
38 Consumers are placed into three classifications (as identified by the
39 Resource::Kind enumeration):
40 
41  - InBound,
42  - OutBound, and
43  - Admin
44 
45  Each caller determines for itself the classification of the Consumer it is
46  creating.

48 ## Resource Loading ##
49 
50 It is expected that a client will impose a higher load on the server
51 when it first connects: the client may need to catch up on transactions
52 it has missed, or get trust lines, or transfer fees. The Manager must
53 expect this initial peak load, but not allow that high load to continue
54 because over the long term that would unduly stress the server.
55 
56 If a client places a sustained high load on the server, that client
57 is initially given a warning message. If that high load continues
58 the Manager may tell the heavily loaded server to drop the connection
59 entirely and not allow re-connection for some amount of time.
60 
61 Each load is monitored by capturing peaks and then decaying those peak
62 values over time: this is implemented by the DecayingSample class.
63 
64 ## Gossip ##
65 
66 Each server in a cluster creates a list of IP addresses of end points
67 that are imposing a significant load. This list is called Gossip, which
68 is passed to other nodes in that cluster. Gossip helps individual
69 servers in the cluster identify IP addreses that might be unduly loading
70 the entire cluster. Again the recourse of the individual servers is to
71 drop connections to those IP addresses that occur commonly in the gossip.
