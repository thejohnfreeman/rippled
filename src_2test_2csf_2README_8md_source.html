1 # Consensus Simulation Framework
2 
3 The Consensus Simulation Framework is a set of software components for
4 describing, running and analyzing simulations of the consensus algorithm in a
5 controlled manner. It is also used to unit test the generic Ripple consensus
6 algorithm implementation. The framework is in its early stages, so the design
7 and supported features are subject to change.
8 
9 ## Overview
10 
11 The simulation framework focuses on simulating the core consensus and validation
12 algorithms as a [discrete event
13 simulation](https://en.wikipedia.org/wiki/Discrete_event_simulation). It is
14 completely abstracted from the details of the XRP ledger and transactions. In
15 the simulation, a ledger is simply a set of observed integers and transactions
16 are single integers. The consensus process works to agree on the set of integers
17 to include in the next ledger.
18 
19 ![CSF Overview](./csf_overview.png "CSF Overview")
20 
21 The diagram above gives a stylized overview of the components provided by the
22 framework. These are combined by the simulation author into the simulation
23 specification, which defines the configuration of the system and the data to
24 collect when running the simulation. The specification includes:
25 
26 - A collection of [`Peer`s](./Peer.h) that represent the participants in the
27  network, with each independently running the consensus algorithm.
28 - The `Peer` trust relationships as a `TrustGraph`. This is a directed graph
29  whose edges define what other `Peer`s a given `Peer` trusts. In other words,
30  the set of out edges for a `Peer` in the graph correspond to the UNL of that
31  `Peer`.
32 - The network communication layer as a `BasicNetwork`. This models the overlay
33  network topology in which messages are routed between `Peer`s. This graph
34  topology can be configured independently from the `TrustGraph`.
35 - Transaction `Submitter`s that model the submission of client transactions to
36  the network.
37 - `Collector`s that aggregate, filter and analyze data from the simulation.
38  Typically, this is used to monitor invariants or generate reports.
39 
40 Once specified, the simulation runs using a single `Scheduler` that manages the
41 global clock and sequencing of activity. During the course of simulation,
42 `Peer`s generate `Ledger`s and `Validation`s as a result of consensus,
43 eventually fully validating the consensus history of accepted transactions. Each
44 `Peer` also issues various `Event`s during the simulation, which are analyzed by
45 the registered `Collector`s.

47 ## Example Simulation
48 Below is a basic simulation we can walk through to get an understanding of the
49 framework. This simulation is for a set of 5 validators that aren't directly
50 connected but rely on a single hub node for communication.
51 
52 ![Example Sim](./csf_graph.png "Example Sim")
53 
54 Each Peer has a unique transaction submitted, then runs one round of the
55 consensus algorithm.
56 
57 ```c++
58 Sim sim;
59 PeerGroup validators = sim.createGroup(5);
60 PeerGroup center = sim.createGroup(1);
61 PeerGroup network = validators + center;
62 center[0]->runAsValidator = false;
63 
64 validators.trust(validators);
65 center.trust(validators);
66 
67 using namespace std::chrono;
68 SimDuration delay = 200ms;
69 validators.connect(center, delay);
70 
71 SimDurationCollector simDur;
72 sim.collectors.add(simDur);
73 
74 // prep round to set initial state.
75 sim.run(1);
76 
77 // everyone submits their own ID as a TX and relay it to peers
78 for (Peer * p : validators)
79  p->submit(Tx(static_cast<std::uint32_t>(p->id)));
80 
81 sim.run(1);
82 
83 std::cout << (simDur.stop - simDur.start).count() << std::endl;
84 assert(sim.synchronized());
85 ```

87 ### `Sim` and `PeerGroup`
88 
89 ```c++
90 Sim sim;
91 PeerGroup validators = sim.createGroup(5);
92 PeerGroup center = sim.createGroup(1);
93 PeerGroup network = validators + center;
94 center[0]->runAsValidator = false;
95 
96 ```
97 
98 The simulation code starts by creating a single instance of the [`Sim`
99 class](./Sim.h). This class is used to manage the overall simulation and
100 internally owns most other components, including the `Peer`s, `Scheduler`,
101 `BasicNetwork` and `TrustGraph`. The next two lines create two differ
102 `PeerGroup`s of size 5 and 1 . A [`PeerGroup`](./PeerGroup.h) is a convenient
103 way for configuring a set of related peers together and internally has a vector
104 of pointers to the `Peer`s which are owned by the `Sim`. `PeerGroup`s can be
105 combined using `+/-` operators to configure more complex relationships of nodes
106 as shown by `PeerGroup network`. Note that each call to `createGroup` adds that
107 many new `Peer`s to the simulation, but does not specify any trust or network
108 relationships for the new `Peer`s.
109 
110 Lastly, the single `Peer` in the size 1 `center` group is switched from running
111 as a validator (the default) to running as a tracking peer. The [`Peer`
112 class](./Peer.h) has a variety of configurable parameters that control how it
113 behaves during the simulation.

115 ## `trust` and `connect`
116 
117 ```c++
118 validators.trust(validators);
119 center.trust(validators);
120 
121 using namespace std::chrono;
122 SimDuration delay = 200ms;
123 validators.connect(center, delay);
124 ```
125 
126 Although the `sim` object has accessible instances of
127 [TrustGraph](./TrustGraph.h) and [BasicNetwork](./BasicNetwork.h), it is more
128 convenient to manage the graphs via the `PeerGroup`s. The first two lines
129 create a trust topology in which all `Peer`s trust the 5 validating `Peer`s. Or
130 in the UNL perspective, all `Peer`s are configured with the same UNL listing the
131 5 validating `Peer`s. The two lines could've been rewritten as
132 `network.trust(validators)`.
133 
134 The next lines create the network communication topology. Each of the validating
135 `Peer`s connects to the central hub `Peer` with a fixed delay of 200ms. Note
136 that the network connections are really undirected, but are represented
137 internally in a directed graph using edge pairs of inbound and outbound connections.

139 ## Collectors
140 
141 ```c++
142 SimDurationCollector simDur;
143 sim.collectors.add(simDur);
144 ```
145 
146 The next lines add a single collector to the simulation. The
147 `SimDurationCollector` is a a simple example collector which tracks the total
148 duration of the simulation. More generally, a collector is any class that
149 implements `void on(NodeID, SimTime, Event)` for all [Events](./events.h)
150 emitted by a Peer. Events are arbitrary types used to indicate some action or
151 change of state of a `Peer`. Other [existing collectors](./collectors.h) measure
152 latencies of transaction submission to validation or the rate of ledger closing
153 and monitor any jumps in ledger history.
154 
155 Note that the collector lifetime is independent of the simulation and is added
156 to the simulation by reference. This is intentional, since collectors might be
157 used across several simulations to collect more complex combinations of data. At
158 the end of the simulation, we print out the total duration by subtracting
159 `simDur` members.
160 
161 ```c++
162 std::cout << (simDur.stop - simDur.start).count() << std::endl;
163 ```

165 ## Transaction submission
166 
167 ```c++
168 // everyone submits their own ID as a TX and relay it to peers
169 for (Peer * p : validators)
170  p->submit(Tx(static_cast<std::uint32_t>(p->id)));
171 ```
172 
173 In this basic example, we explicitly submit a single transaction to each
174 validator. For larger simulations, clients can use a [Submitter](./submitters.h)
175 to send transactions in at fixed or random intervals to fixed or random `Peer`s.
176 
177 ## Run
178 
179 The example has two calls to `sim.run(1)`. This call runs the simulation until
180 each `Peer` has closed one additional ledger. After closing the additional
181 ledger, the `Peer` stops participating in consensus. The first call is used to
182 ensure a more useful prior state of all `Peer`s. After the transaction
183 submission, the second call to `run` results in one additional ledger that
184 accepts those transactions.
185 
186 Alternatively, you can specify a duration to run the simulation, e.g.
187 `sim.run(10s)` which would have `Peer`s continuously run consensus until the
188 scheduler has elapsed 10 additional seconds. The `sim.scheduler.in` or
189 `sim.scheduler.at` methods can schedule arbitrary code to execute at a later
190 time in the simulation, for example removing a network connection or modifying
191 the trust graph.
